# Basketball PBP Augmentation — Status Report

## What we are trying to do

Augment a partial digital play-by-play (PBP) from a competition scoring system
into a more complete statistical record of a basketball game. The scoring system
captures made baskets and fouls but not missed shots, rebounds, substitutions,
or shot locations. The goal is to use video footage and AI to detect and add
these missing events.

The output format is [BGDL](../BGDL.md) — a custom open specification developed alongside this
project for describing basketball game events, supporting both named court regions
and absolute x,y coordinates for shot locations.

The intended users are coaching staff at a community basketball club, across
multiple age groups / divisions in a high performance rep program.

---

## Data sources available

**Competition scoring system (primary)**
- Made baskets with player number and approximate game clock
- Fouls with player number
- Team rosters with jersey numbers
- Accessed via API with existing Python extraction code and database

**Human stats service (ground truth for evaluation)**
- Complete PBP including missed shots, rebounds, assists, turnovers
- Currently used for the top division team at ~$30/game
- Output used as ground truth to evaluate automated pipeline accuracy

---

## Video sources available

**In-stadium roof-mounted camera with automatic panning (free)**
- Fixed high sideline position, good court angle
- 720p resolution, automated ball-following pan
- Available for approximately half the competition venues
- Limitations: poor ball-following when adjacent court is active, 720p
  insufficient for jersey OCR, automated pan sometimes follows wrong ball

**High-end camcorder operated from sideline (owned)**
- 4K resolution, approximately one-third court framing
- Operated by the project owner following play, generally with wide framing
- Strong optical stabilisation, good audio track
- Limitation: ~53% of frames are panning, partial court framing loses
  3pt arc context

**Compact action cameras x3 (owned)**
- Intended for fixed placement behind each baseline
- Rebound detection from behind-basket angle
- Placement constrained by gym layout — not yet integrated into pipeline

## Pipeline architecture

### Data ingestion layer (built, working)

- Competition API extraction to database
- BGDL conversion from competition data
- Paper scoresheet OCR path (designed, not yet built)

### Core pipeline (POC built, being iterated)

```
Competition PBP (partial BGDL)
        +
Video footage
        ↓
Sync: align video timestamps to competition PBP events
        ↓
Event detection: find missed shots between confirmed makes
        ↓
Attribution: assign events to teams and players
        ↓
Location: estimate court coordinates
        ↓
Augmented BGDL output
```

### Current implementation approach

- Python, running on Apple Silicon laptop
- Claude Haiku vision API for frame analysis
- Async API calls (10 concurrent) for throughput
- BGDL output with confidence scores and source annotations

## What has been built and tested

### POC v1 — roof-mounted camera with automatic panning

**Approach:** Sample frames at 1fps, send every frame to Claude vision API
asking "is there a shot attempt, what team, what jersey number, what are
the coordinates."

**Results (one quarter, 25 ground truth missed shots):**

| Metric | Result |
| --- | --- |
| Recall | 48% (12/25) |
| Precision | 24% (12/50 detected) |
| Team correct | 75% |
| Shot type correct | 83% |
| Jersey correct | 8% |
| Avg time offset | 2.6s |
| Cost | ~$2.91 for full game |

**Key findings:**
- Shot type classification via ref signal + foot position works well at 83%
- Jersey OCR completely unreliable at 720p — fundamental resolution limitation
- Precision is the dominant problem — 76% of detected events are false positives
- Sync failed visually, required manual offset

---

### POC v2 — high-end camcorder operated from sideline

**Changes made:** Added optical flow frame classification
(stable/panning/transitioning), raised confidence threshold from 0.5 to 0.65,
re-enabled jersey OCR, updated camera context in vision prompt.

**Results (one quarter, 22 ground truth missed shots):**

| Metric | High-end camcorder | Roof-mounted camera |
| --- | --- | --- |
| Recall | 59% | 48% |
| Precision | 23% | 24% |
| Team correct | 46% | 75% |
| Shot type correct | 46% | 83% |
| Jersey correct | 0% | 8% |
| Avg time offset | 2.5s | 2.6s |

**Key findings:**
- Recall improved slightly — 4K footage detects more real shots
- Precision unchanged at ~23% — false positives are high-confidence wrong
  detections, not low-confidence ones. Raising the threshold does not help
- Team and shot type both collapsed — partial court framing means the 3pt arc
  and attacking direction context are often not visible
- Jersey OCR still broken — 53% of frames are panning, stable frames still
  returned 0/5 correct jersey reads
- The confidence threshold change had no measurable effect on precision

## Current diagnosis

**The core problem is false positives from single-frame vision LLM detection**

Claude vision sees one frame and pattern-matches on things that look like
basketball actions. Without temporal context (ball trajectory across multiple
frames), it cannot distinguish a real shot arc from a player jumping to contest
a pass, a player reaching for a rebound, or an incidental arm raise. Raising
the confidence threshold doesn't help because these false positives are
generated with high confidence.

**This is an architectural problem, not a prompting problem**

Further prompt iteration is unlikely to move precision meaningfully. The vision
LLM is the wrong tool for bulk shot detection. It is the right tool for:
- Confirming a shot attempt identified by a ball tracker
- Ref signal detection
- Court coordinate estimation
- Jersey OCR on stable high-resolution frames

**Jersey OCR requires stable frames at the right moment**

On both camera types, jersey numbers are not readable at the moment of shot
detection — either due to motion blur (panning) or the player not facing the
camera. A separate jersey identification pass on stable frames away from the
shot moment may be more productive.

**Team detection needs jersey colour**

Partial court framing makes attacking direction inference unreliable. Using
jersey colour from game metadata is a simple fix that does not require court
geometry.

## What is being built next

**Ball tracker test (in progress)**

Local YOLOv8 ball detection + ByteTrack trajectory analysis on high-end
camcorder footage. No API calls. Identifies shot arc candidates from ball
trajectory (upward arc toward basket across multiple frames). Outputs annotated
video and candidate timestamps for manual evaluation against ground truth.

This answers the key question: can ball tracking reliably identify shot attempt
candidates on this footage, and does it produce fewer false positives than
vision LLM frame sampling?

**Jersey colour team detection (in progress)**

Small targeted update to use jersey colours from game metadata for team
attribution. Addresses the team ID collapse on partial-court footage.

---

## Planned subsequent steps

**If ball tracker performs well:**
Integrate into main pipeline as the shot detection layer. Vision LLM demoted
to confirmation and attribution only. Expected outcome: precision improves
from ~23% to 50%+, API cost drops significantly.

**Baseline cameras integration:**
Once single-camera pipeline is producing acceptable precision, add the two
compact action cameras behind each baseline. Ball-through-net detection gives
unambiguous made/missed classification and rebound detection — the hardest
remaining problems for single-camera approaches.

**Roof-mounted camera as supplementary source:**
Treat roof-mounted footage as a free supplementary input for games where the
project owner is not present to film. Lower accuracy ceiling but zero effort —
useful for teams who cannot justify the cost of the human stats service.

## Target accuracy (not yet achieved)

| Metric | Current best | Target |
| --- | --- | --- |
| Precision | 23% | 50%+ |
| Recall | 59% | 70%+ |
| Team correct | 75% (roof-mounted) | 85%+ |
| Shot type correct | 83% (roof-mounted) | 80%+ |
| Jersey correct | 8% | 40%+ |

Achieving target accuracy would produce a PBP augmentation useful for shot
volume and tendency analysis at the team level, and directionally useful at
the individual player level. It would not replace the human stats service for
the top division team, but would provide meaningful value to lower division
teams who currently receive no stats at all.
